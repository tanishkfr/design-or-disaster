Overall

The motion system is intentional and mostly correct — custom easing curves, animation-fill-mode patterns, and well-considered timing for an editorial portfolio. The big gap is a single accessibility bug in the reduced-motion handler that would cause layout flashes for vestibular users. Critical Jakub Accessibility

Reduced-motion handler causes layout flash and leaves transform motion active

What The @media (prefers-reduced-motion: reduce) block in tokens.css sets transition-property: opacity !important. This forces all CSS transitions to only transition opacity — meaning elements that start at translateY(12px) (e.g. JurorEntry's .entry) will instantly jump to their final position while fading in. Additionally, keyframe animations (enterUp, pageEnter, closingEnter, blockSlideUp) still run at 150ms with full transform motion — the standard is near-zero (0.01ms) to make them instant.

Why it matters Users who enable reduced motion often have vestibular disorders. A layout flash (opacity transition while position jumps) is exactly the kind of sudden movement this setting exists to prevent. JurorEntry's stagger sequence — where 5+ cards each jump into position — would be particularly disorienting.

Recommended fix Replace the reduce block with the standard pattern. Remove transition-property: opacity !important and set both animation and transition durations to 0.01ms. This makes all keyframes effectively instant (final state snaps in immediately) while preserving layout. Elements using CSS transitions with transforms will also snap correctly — no position flash. 

@media (prefers-reduced-motion: reduce) {
  *,
  *::before,
  *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}

Also update the :root token overrides to set all durations to 1ms (near-zero) so any JS reading the token also gets the correct reduced-motion value.

Important Jakub

View transitions snap — components exit instantly before the next view enters

What When App.jsx switches between 'archive', 'casefile', 'verdictchamber', 'designeye', and 'finalreport', the unmounting component disappears instantly. Each incoming component has an enter animation, but the outgoing one is cut — creating a hard visual jump before the new view fades/slides in.

Why it matters Jakub's principle: exits should be subtler than enters, but they should exist. A hard cut followed by an enter animation is rhythmically wrong — it creates a flash of empty screen. The editorial register of this project (document feels, case transitions) makes the cut read as abrupt rather than intentional.

Recommended fix Add a short exit fade to the outgoing view before swapping state. The simplest approach: introduce a short exiting state in App.jsx — set it on navigation, wait one tick for the CSS transition to fire, then swap the view state. A subtle opacity + translateY(-6px) exit at 200ms (subtler than the 600ms enters) creates rhythm without requiring AnimatePresence. 

/* Add to each top-level view component */
.viewExit {
  opacity: 0;
  transform: translateY(-6px);
  transition:
    opacity 200ms var(--ease-in),
    transform 200ms var(--ease-in);
  pointer-events: none;
}

Important Emil

No :active scale feedback on verdict cards, evidence tags, or confidence options

What Verdict cards (.verdictCard), evidence tags (.tag), and confidence options (.confidenceOption) have hover states but no :active scale-down feedback. Users click these elements multiple times per case — they're the core interaction mechanism.

Why it matters Emil's rule #1: subtle scale(0.97) on :active creates immediate tactile feedback. Without it, clicking feels flat — the UI doesn't respond physically. For tags specifically (multiple clicks per case, high frequency within a session), the absence is noticeable. The animation is instant on press and instant on release — it doesn't add duration, it adds feel.

Recommended fix Add to all three selectors. Use transform transition at --duration-fast (150ms) which is already in the transition list for tags. For cards, append to the existing transition property list: 

/* SubmissionPanel.module.css */
.tag:active,
.verdictCard:active {
  transform: scale(0.97);
}

.confidenceOption:active {
  transform: scale(0.98); /* smaller movement for a text element */
}

07 · Lens Reference

Lens referenced most: Jakub Krehel — Production Polish. This is a creative portfolio designed to be experienced repeatedly (job applications, portfolio reviews). Jakub's invisible-quality philosophy — motion that you feel without noticing — is the right primary lens for a project where the content is the design criticism, not the UI chrome.

To lean more Emil: Question whether the 600ms enter durations on view-level components (Archive, VerdictChamber, FinalReport) could drop to 400ms. They're occasional interactions — 300–400ms is Emil's comfort zone for daily use.
To lean more Jhey: The DesignEyeIntro headline is the one place where a more expressive entrance technique (clip-path reveal, @property animation, letter-spacing collapse) would genuinely reward the viewer. The register allows it at that specific pivot moment.
To lean more Jakub: Implement exit transitions on view changes. This is the production-polish gap that Jakub would consider unshipped — the enters are there, the exits are missing, and a polished product needs both.
Priority if one fix only: The reduced-motion bug. It's the only accessibility issue, it's fast to fix, and it's the kind of thing that signals production readiness (or its absence) to technical reviewers.
